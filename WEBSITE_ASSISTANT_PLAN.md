## Философия: единая система — многообразие экспертизы

**Это новая архитектура экспертных систем 21 века.**

Экспертные системы 20 века были монолитны: одна область, один язык, один эксперт
закодировал правила — система умерла вместе с ним.

Новая архитектура строится на двух принципах:
- **Единая система** — универсальный контур: база знаний, расчётное ядро,
  агент, обратная связь, самообучение. Архитектура не меняется.
- **Многообразие экспертизы** — содержимое меняется под любую область.
  Трансформаторы, химия, авиация, медицина — одна платформа, разные базы знаний.

```
        ЕДИНАЯ АРХИТЕКТУРА
┌───────────────────────────────┐
│  База знаний (DSL + pgvector) │
│  Расчётное ядро (SymPy/SciPy) │
│  Агентный оркестратор         │
│  Библиотека визуалов          │
│  Контур обратной связи        │
└──────────┬────────────────────┘
           │ наполняется
    ┌──────┴──────┐
    ▼             ▼
Трансформаторы   Химпроцессы
Авиация          Атомная энергетика
Металлургия      Нефтегаз  ...
```

**Разделение труда:**

```
Кибернетик (автор платформы)     Эксперт-предметник
─────────────────────────────────────────────────────────
Архитектура обратных связей      Формулы и ограничения
Структура базы знаний            Данные в эту структуру
Механизм самообучения            Валидация результатов
Агентная оркестрация             Экспертные суждения
DAG-решатель                     Граф зависимостей области
```

**Контур управления:**

```
Вход:  запрос инженера
          ↓
       агент + база знаний + расчётное ядро
          ↓
Выход: результат
          ↓
       Обратная связь: принял / отверг / уточнил
          ↓
       Коррекция базы знаний
```

**Ценность для корпорации** — не программа для расчётов, а институциональная память
которая не уходит на пенсию. Экспертиза тиражируется, сохраняется и передаётся
следующему поколению без потерь.

**Рынок:**

```
Было: ПО для трансформаторных заводов → десятки компаний
Стало: платформа усиления экспертизы  → тысячи предприятий:
       атомная энергетика, химия, авиация, оборонка, нефтегаз, металлургия
```

Трансформаторная вертикаль — reference implementation.
Доказал на одной нише → тиражируешь архитектуру на любую другую.

## Идея

Сайт-ассистент — не лендинг и не форма. Это агентная платформа специалиста:
справочник, калькулятор, приёмник данных и генератор страниц в одном интерфейсе.

Первая вертикаль: **проектирование силовых трансформаторов большой мощности**.

Причины выбора вертикали:
- Чёткая логика, проверенные теоремы и законы — нет галлюцинаций
- Существующие инструменты: Фортран 80-90х, RALE (до 10 МВА), TDS (заброшен 2017)
- Ниша 10–600 МВА практически пуста
- Производственные данные закрыты у крупных игроков — возможность калибровки
- Рынок $38+ млрд к 2030, дефицит инженеров-конструкторов нарастает

## Схема базы данных (ядро знаний)

```sql
-- Концепты (атомарные единицы знания)
concepts     id, domain, category, symbol,
             labels:jsonb,       -- {ru: "...", en: "...", de: "..."}
             description:jsonb

-- Формулы (параметрическая модель = строка здесь)
formulas     id, concept_id,
             expression,         -- SymPy-совместимое выражение
             inputs:jsonb,       -- [{symbol, unit, type}, ...]
             outputs:jsonb,
             source_standard,    -- "IEC 60076-1" / "ГОСТ 3484"
             source_clause,      -- "п. 5.3.2"
             verified:bool

-- Параметры
parameters   id, formula_id, symbol, unit, type, min, max, default

-- Модели (граф зависимостей = набор формул + порядок)
models       id, name, domain,
             dag:jsonb           -- граф зависимостей, параметрическая модель

-- Нормативы
standards    id, code, edition, language, domain

-- UI-блоки (библиотека визуалов)
ui_blocks    id, type,
             schema:jsonb,       -- структура входов/выходов
             template:jsonb      -- шаблон рендера

-- Сессии (основа самообучения)
sessions     id, user_id, queries:jsonb, gaps:jsonb, model_used
```

**Главный принцип:** знание живёт в БД. Код только исполняет.
Добавить новую формулу = добавить строку, не деплоить.

## Внутренний язык модели (DSL / онтология)

Все концепты хранятся в языконезависимом виде:

```yaml
concept: BEAM_SIMPLY_SUPPORTED
  domain: structural_mechanics
  properties:
    L: { type: Length,         unit: m,   symbol: L }
    q: { type: LinearLoad,     unit: N/m, symbol: q }
    E: { type: ElasticModulus, unit: Pa,  symbol: E }
    I: { type: MomentOfInertia,unit: m⁴,  symbol: I }
  derived:
    M_max: { expr: "q*L**2/8",         label: {ru: "Макс. момент", en: "Max moment"} }
    f_max: { expr: "5*q*L**4/384/E/I", label: {ru: "Прогиб",      en: "Deflection"} }
  source: { ru: "СП 20.13330", intl: "ISO 2394" }
```

Правила мультиязычности:
- Внутренние ID всегда на английском (`concept_id: CORE_FLUX_DENSITY`)
- Отображение — через `labels:jsonb` на языке пользователя
- Поиск — через multilingual-e5-large эмбеддинги (pgvector)
- Числа и единицы — через Pint + locale
- Топонимы и имена — через стандартизированные ID, не строки

## Агентная архитектура

```
Пользователь (любой язык)
        ↓
   Intake Agent       — квалифицирует запрос, собирает контекст
        ↓
   Search Agent       — ищет концепты и формулы в БД (3 уровня иерархии)
        ↓
   Calc Agent         — исполняет DAG-модель через SymPy
   Doc Agent          — генерирует документ / форму
   Page Agent         — собирает страницу из UI-блоков
        ↓
   Memory Agent       — сохраняет контекст сессии
```

## Поэтапное проектирование

### Этап 1 — Ядро знаний
- Схема БД + API CRUD
- DSL для концептов и формул (YAML → БД)
- Загрузчик знаний
- Первые 30–50 концептов: трансформаторостроение
- Мультиязычность с нуля: ru / en

### Этап 2 — Расчётный движок
- SymPy исполняет формулы из БД (не из кода)
- Pint валидирует единицы автоматически
- DAG-решатель: каскадный пересчёт при изменении параметра
- API: `POST /calculate {model_id, params}` → results

### Этап 3 — Поиск и навигация
- pgvector эмбеддинги для всех концептов
- Трёхуровневый иерархический поиск
- Мультиязычная модель (multilingual-e5-large)
- Семантический поиск: "потери холостого хода" = "no-load losses" = "Leerlaufverluste"

### Этап 4 — AI-агент
- LLM как интерфейс к поисковику и движку
- Агент квалифицирует запрос → выбирает модель → считает
- Контекст сессии: помнит историю запросов
- Трассировка: "почему такой результат" → цепочка формул + источники

### Этап 5 — Библиотека визуалов
- Компоненты: CalcBlock, FormulaCard, ParameterSweep, ResultSummary, StepForm
- Страница генерируется агентом из блоков по запросу
- KaTeX для формул, D3.js для графиков, Three.js для 3D-моделей

### Этап 6 — Самообучение
- Логируем: что спрашивают, на что система не смогла ответить (`gaps`)
- Dashboard для эксперта: незакрытые вопросы → он докладывает базу
- Производственные данные → калибровка моделей (Этап после получения доступа)
- A/B результатов: какой ответ пользователь принял/отверг

## Стратегия убеждения технического директора

**Его позиция:** "AI не знает то, что знаю я"

**Ответ:** не спорить — показать что система делает то, что Фортран **не может**:

| Фортран 80-х | Платформа |
|---|---|
| Фиксированный расчёт | Параметрический DAG |
| Меняешь → пересчитываешь вручную | Меняешь → всё обновляется каскадно |
| Нет трассировки | Каждая формула = ссылка на ГОСТ/IEC |
| Один вариант | Параметрический свип → оптимум |
| Нет визуализации | Графики, сечения, зависимости |
| Один человек знает | База знаний доступна всей команде |

**Ключевой аргумент:** "Твои формулы уже в системе — с твоими ссылками,
твоими ограничениями. Ты не заменён — ты тиражирован."

**Дорожка к производственным данным:**
1. Демо на открытых расчётах → убеждаем ТД
2. Загружаем исторические ТУ и протоколы испытаний
3. Калибруем модели на реальных данных завода
4. Платформа знает "как у нас принято" — экспертиза ТД в системе

## Ключевое конкурентное преимущество: SVG → TDTP XML → БД

Существующая возможность в tdtp-framework: конвертация SVG-схем в TDTP XML
с последующей работой со структурой как с базой данных.

**Что это даёт платформе:**

```
Инженер рисует схему (SVG)
         ↓
  SVG → TDTP XML         ← уже работающий конвертер
         ↓
  XML как БД             ← запросы к структуре схемы
         ↓
  Автоматическое заполнение параметрической модели
         ↓
  Расчёт + результат
         ↓
  Обновлённая схема (SVG) ← цикл замкнулся
```

Это двунаправленный мост между чертежом и расчётом — его нет ни у RALE, ни у JMAG,
ни у Фортран-программ.

**Прецедент:** со схемами БД (ER-диаграммы → TDTP XML) подход сработал отлично
благодаря устоявшимся стандартам нотации.

## Область исследования: стандартизация схем трансформаторов

### Проблема

Для схем БД существуют единые стандарты нотации (ERD, DDL), поэтому парсинг
предсказуем. Для трансформаторных схем единого стандарта нет — это усложняет
автоматическое извлечление параметров из SVG.

### Что существует

| Стандарт | Охват | Применимость |
|---|---|---|
| IEC 60617 | Графические символы для электросхем | Частично — символы есть, параметры нет |
| IEC 61850 SCL | XML для автоматизации подстанций | Близко, но для SCADA, не для конструирования |
| IEC 61968/61970 CIM | Общая информационная модель энергосистем | Слишком высокий уровень абстракции |
| ГОСТ 2.755 | Обозначения для электросхем (СНГ) | Отличается от IEC — параллельный мир |
| ANSI/IEEE Std 315 | Символы для США | Третий диалект |

### Ключевой вывод

Единого стандарта для **обмоточных схем** и **конструктивных чертежей**
трансформаторов не существует. Каждый производитель использует свои обозначения.

### Варианты решения

**Вариант A — Собственный промежуточный формат (рекомендуется для старта)**
```
Определить минимальный набор примитивов для трансформаторной схемы:
- Обмотка (ВН/НН/СН): витки, провод, изоляция
- Сердечник: тип, сечение, материал
- Отводы и РПН
- Узлы соединений

SVG → парсер → внутренний формат → TDTP XML → БД
```
Плюс: полный контроль, быстрый старт
Минус: нет совместимости с внешними инструментами

**Вариант B — Расширение IEC 61850 SCL**
```
Взять существующий XML-стандарт, добавить элементы для конструктивных параметров
```
Плюс: совместимость с энергетической отраслью
Минус: стандарт не предназначен для конструирования, большой overhead

**Вариант C — Граф-онтология поверх SVG**
```
Не парсить SVG напрямую, а обучить модель распознавать элементы схемы
и маппить на концепты из БД знаний
```
Плюс: работает с любым входным форматом
Минус: требует обучающих данных, сложнее верифицировать точность

## Перспектива: Моделирование поведения *(при одобрении проекта)*

Трансформатор — это система связанных нелинейных дифференциальных уравнений
в открытой среде. Это открывает следующий уровень платформы: от расчёта к симуляции.

### Почему система нелинейная и открытая

```
Электромагнитная подсистема
  dΨ/dt = u - Ri        нелинейная: Ψ = f(i) через кривую B-H
  + гистерезис          память системы, путь-зависимость

Тепловая подсистема
  C·dT/dt = P_loss - P_cooling   нелинейная: конвекция, вязкость масла

Механическая подсистема
  F_sc = f(i²)          силы КЗ — удар, а не квазистатика

Связи между подсистемами:
  температура → сопротивление → потери → температура (цикл)
  насыщение → форма тока → силы → вибрация → шум
```

Система открытая: питается от сети с гармониками, нагрузка непредсказуема,
внешняя температура влияет на охлаждение, режим нейтрали меняет поля.

### Фазы симуляции

| Фаза | Содержание | Зависимость |
|---|---|---|
| Этап 7 | Квазистатика: нагрузочные и температурные характеристики | База этапов 1–6 |
| Этап 8 | Динамика: переходные процессы, бросок тока, грозовой импульс | + производственные данные |
| Этап 9 | Digital Twin: реальные измерения + модель в реальном времени | + доступ к телеметрии |

### Стек для симуляции

```python
# Этапы 7–8: нелинейные ОДУ
scipy.integrate.solve_ivp        # умеренная нелинейность
Julia DifferentialEquations.jl   # жёсткие системы (stiff) — трансформатор именно такой

# Этап 9: Digital Twin
# реальные измерения → идентификация параметров → коррекция модели в реальном времени
```

### Ключевая зависимость от производственных данных

Сердце нелинейности — кривая B-H конкретной марки электротехнической стали.
Это не формула, а таблица измерений. У каждого завода своя сталь, свои замеренные
характеристики. Именно здесь закрытые производственные данные становятся
незаменимыми и превращаются в конкурентное преимущество платформы.
