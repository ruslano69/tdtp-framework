TDTP CLI - Table Data Transfer Protocol Command Line Interface
Version: {VERSION}

USAGE:
  tdtpcli [command] [options]

COMMANDS:

  Database Operations:
    --list[=pattern]           List tables; filter by glob (e.g. --list=user*, --list=order?)
    --export <table>           Export table to TDTP XML file
    --import <file>            Import TDTP XML file to database
    --inspect-table <table>    Inspect live DB table: native types, FKs, row count, sample row

  File Operations:
    --test <tdtp-file>         Dry-run integrity check: decompress in memory, verify XXH3 checksum,
                               validate XML, count rows vs header (no DB connection needed)
    --inspect <tdtp-file>      Print YAML metadata summary (no config needed)
    --to-csv <tdtp-file>       Convert TDTP file to CSV. Handles compressed (zstd/kanzi),
                               compact v1.3.1, and v1.4 integrity packets.
    --to-html <tdtp-file>      Convert TDTP to HTML viewer (fast preview)
    --diff <file-a> <file-b>   Compare two TDTP files and show differences
    --merge <files>            Merge multiple TDTP files into one
    --to-compact <tdtp-file>   Convert existing TDTP v1.x file to compact v1.3.1 format

  XLSX Operations:
    --to-xlsx <tdtp-file>      Convert TDTP XML to XLSX
    --from-xlsx <xlsx-file>    Convert XLSX to TDTP XML
    --export-xlsx <table>      Export table directly to XLSX
    --import-xlsx <xlsx-file>  Import XLSX directly to database

  Message Broker Operations:
    --export-broker <table>    Export table to message broker
                               All packets sent in a single network roundtrip (SendBatch).
                               Compression and XML serialization run in parallel goroutines.
    --import-broker            Import from message broker
                               Receives all packets first, decompresses in parallel, imports in order.
    --import-broker --output   Save received packets as TDTP files instead of importing to DB.
                               Multi-part exports are saved as base_part_N_of_Total.tdtp.xml
    --import-broker --raw      Save broker messages as-is (no parse, no decompress, no validate).
                               Reads TotalParts from the first message header for correct naming.
                               Useful for forwarding or archiving raw queue traffic.
    --import-broker --keep     Non-atomic mode: import each part immediately as it arrives.
                               If a later part fails, earlier parts stay committed in the table.
                               Use for batches too large for a single DB transaction.
                               Default (no --keep): all parts in one transaction — all-or-nothing.
    --listen                   Streaming consumer daemon (Kafka only)
                               Listens to Kafka topic and imports data as stream parts arrive.
                               Requires stable channel (99.99%+ uptime). NOT for RabbitMQ/MSMQ.

    Security note: queue/topic name is always taken from the config file, never
    from the command line. This prevents CLI users from redirecting exports to
    an arbitrary queue and intercepting sensitive packets. The operator controls
    the destination; the user controls what data to export (--export-broker <table>).

    Broker tiers:
      MSMQ     — Legacy     (Windows-only, no ordering guarantees, batch mode only)
      RabbitMQ — Stability  (reliable delivery, batch mode)
      Kafka    — Speed      (ordered partitions, batch + --listen streaming; production-ready)

  Incremental Sync:
    --sync-incremental <table> Incremental sync from table

  ETL Pipeline:
    --pipeline <file>          Execute ETL pipeline from YAML config
    @name=value                Pass variable to pipeline (any number, after --pipeline)
                               Quotes around value are stripped automatically: @dept="97-256" → 97-256
                               Used variables are embedded in the output packet as PipelineContext
    --expect-var <name=value>  Verify PipelineContext variable before import (repeatable)
                               Import fails before any DB writes if value doesn't match

OPTIONS:

  General:
    --config <file>            Configuration file (default: config.yaml)
    --output <file>            Output file path
    --table <name>             Target table name (overrides name from XML during import)
    --strategy <name>          Import strategy: replace, ignore, fail, copy
    --batch <size>             Batch size for bulk operations (default: 1000)
    --readonly-fields          Include read-only fields (timestamp, computed, identity)

  Compression:
    --compress                 Enable compression. XXH3-64 checksum of the compressed blob is added
                               automatically and verified on --test, --import, --to-csv, --to-html.
    --compress-algo <algo>     Algorithm: zstd (default) or kanzi (4× denser than raw, 30% denser than zstd)
    --compress-level <n>       Compression level: 1-19 (zstd) or 6-7 (kanzi), default: 3
    --packet-size <MB>         Max broker packet size in MB (default 0 = ~1.9MB; use 8 for kanzi)
    --fast                     Skip NULL/NaN/Inf detection for maximum throughput (no schema markers)
    --fallback-row-limit <n>   Max rows loaded into memory when SQL pushdown fails (default: 1000000)
                               Protects production DBs from accidental full-table scans; 0 = unlimited

  v1.4 Integrity:
    --integrity                Stamp packet with TDTP v1.4 xxh3_128 hashes: Schema + Data + Packet
                               fingerprint. Hash covers plain-text rows BEFORE compression.
                               Upgrades packet version to 1.4. Combine with --compress for full protection.
    --mercury-url <url>        Register packet fingerprint in xzMercury after export
                               (e.g. http://mercury:3000). Empty = local integrity only.
    --mercury-caller <name>    X-Caller identity sent to xzMercury (default: "tdtpcli")
                               Use service-account name, e.g. svc-exporter, etl-prod-1

  Compact Format (v1.3.1):
    --compact                  Enable compact format on export: fixed fields written once per group,
                               repeated rows carry only changing values (significant size reduction
                               for wide tables with many repeated values, e.g. denormalized joins)
    --fixed-fields <fields>    Comma-separated fixed field names, or '_' to auto-detect from
                               _prefix columns (e.g. _dept_id, _dept_name → auto-fixed)
    --compact-tail             Write an explicit tail row with all fixed fields for stream validation
                               and carry-state handoff between parts
    --to-compact <file>        Convert an existing TDTP v1.x file to compact v1.3.1 format in-place
                               (output file name set with --output)

  CSV Options (--to-csv):
    --delimiter <sep>          Field separator character (default: ',')
                               Examples: --delimiter ';'  --delimiter '\t'  --delimiter '|'
    -d <sep>                   Shorthand for --delimiter
    --cp <encoding>            Output code page: utf8 (default), 1251 (Windows Cyrillic), 866 (DOS)
    --bom                      Prepend UTF-8 BOM byte sequence (EF BB BF) — helps Excel
                               auto-detect UTF-8 encoding without manual import wizard
    --mercury-url <url>        xzMercury URL for v1.4 packet pre-flight verification (optional)

  TDTQL Filters:
    --where <condition>        WHERE clause (e.g., 'age > 18 AND status = active')
                               Bracket-quoted identifiers for names with spaces or special chars
                               (MSSQL/Access style): --where '[Termination Date] = "1753-01-01"'
                               Brackets are stripped and the name is properly quoted per dialect:
                                 MSSQL → [Termination Date]  PG/MySQL → "Termination Date"
                               Repeatable: --where 'age > 18' --where 'status = active' (AND)
    -w <condition>             Shorthand for --where (repeatable)
    --order-by <fields>        ORDER BY clause (e.g., 'name ASC, age DESC')
    --limit <n>                LIMIT rows: positive = first N, negative = last N (like tail -n)
    -n <n>                     Shorthand for --limit
    --offset <n>               OFFSET number of rows to skip
    --fields <col1,col2,...>   Column projection: export/import/to-csv only listed columns
                               (e.g. 'id,email,status')
                               For --import: whitelist — only these columns are written to DB
                               For --to-csv: only listed columns appear in output CSV
                               For --sync-incremental: tracking field is always included automatically
                               Bracket-quoted names for fields with spaces or commas:
                                 --fields "id,[Birth Date],status"
                                 --fields "[First, Last],[Birth Date]"

  XLSX Options:
    --sheet <name>             Excel sheet name (default: Sheet1)

  Incremental Sync Options:
    --tracking-field <field>   Field to track changes (default: updated_at)
    --checkpoint-file <file>   Checkpoint file (default: checkpoint.yaml)
    --batch-size <size>        Batch size for sync (default: 1000)

  ETL Pipeline Options:
    --unsafe                   Enable unsafe mode (allows all SQL, requires admin)
    --expect-var <name=value>  Require PipelineContext variable to match before import (repeatable)
                               Fails fast before any DB writes on mismatch or missing variable
    --enc                      Encrypt output via xZMercury (AES-256-GCM, UUID-binding)
                               Requires security.mercury_url in pipeline YAML

  Pipeline Variable Substitution (@name=value):
    SQL string context:        WHERE col = '@dept'       → WHERE col = '97-256'
    SQL numeric context:       WHERE year = @year        → WHERE year = 2025
    YAML fields ({{name}}):    destination: "out/{{dept}}.tdtp.xml" → out/97-256.tdtp.xml
    Applies to:                sources[].query, transform.sql, description,
                               output.tdtp.destination, output.xlsx.destination
    Validation:                declared but not provided → error
                               provided but unused       → warning

  Diff Options:
    --key-fields <fields>      Key fields for comparison (comma-separated)
    --ignore-fields <fields>   Fields to ignore in diff (comma-separated)
    --case-sensitive           Case-sensitive comparison (default: false)

  Merge Options:
    --merge-strategy <name>    Merge strategy: union, intersection, left, right, append
                               (default: union)
    --show-conflicts           Show detailed conflict information

  Field Name Sanitization (--import):
    --clear                    Replace special chars in field names with safe SQL identifier tokens.
                               Symbol map: % → _pct, $ → _usd, # → _xh, @ → _at, * → _star,
                                           & → _and, ? → _is, ~ → _not, + → _plus, = → _eq,
                                           ! → _bang, ^ → _hat, < → _lt, > → _gt,
                                           space . , - / \ ` : | ; → _
                               Remaining non-ASCII chars → _. Consecutive __ collapsed.
    --translit                 Transliterate non-ASCII field names to closest ASCII equivalents
                               using go-unidecode before applying --clear rules.
                               Cyrillic: "Имя" → "Imia", "Дата рождения" → "Data_rozhdeniia"
                               European: "Österreich" → "Osterreich", "Ñoño" → "Nono"
                               Original names are preserved as DB column comments:
                                 PostgreSQL: COMMENT ON COLUMN t.col IS 'original: ...'
                                 MySQL:      col TEXT COMMENT 'original: ...'
                               Test XMLs: tests/sanitize/ (access_fields, cyrillic, exotic_mixed)

  Data Processors:
    --mask <fields>            Mask sensitive fields (comma-separated)
    --validate <file>          Validate fields (YAML rules file)
    --normalize <file>         Normalize fields (YAML rules file)

  Configuration:
    --create-config-pg         Create PostgreSQL config template
    --create-config-mssql      Create MS SQL config template
    --create-config-sqlite     Create SQLite config template
    --create-config-mysql      Create MySQL config template

  Misc:
    --version                  Show version information
    --help                     Show this help message

EXAMPLES:

  # List all tables
  tdtpcli --list --config pg.yaml

  # List tables matching a pattern (use = to pass the pattern)
  tdtpcli --list=user* --config pg.yaml
  tdtpcli --list=order? --config pg.yaml
  tdtpcli --list=%log% --config pg.yaml

  # Export table to TDTP XML
  tdtpcli --export users --output users.tdtp.xml

  # Export with filters
  tdtpcli --export orders --where 'status = active' --limit 100

  # Field names with spaces/special chars (MSSQL/Access bracket syntax)
  # Table/field names with $ or spaces — wrap in single quotes + square brackets
  tdtpcli --export '[ZTR$Employee]' --where "[Termination Date] = '1753-01-01'"
  tdtpcli --export orders --where "[Complex Name of Date] = '1999-01-01'"

  # Export only specific columns (column projection)
  tdtpcli --export clients --fields id,email,status --output clients_slim.tdtp.xml

  # Import only specific columns from a wide TDTP file
  tdtpcli --import clients_full.tdtp.xml --fields id,email,status --table clients_slim

  # Export with compression (zstd level 3, balanced)
  tdtpcli --export users --compress --output users.tdtp.xml

  # Export with kanzi (30% denser than zstd, large text tables)
  tdtpcli --export operations_log --compress --compress-algo kanzi --compress-level 6 --output ops.tdtp.xml

  # Export with compression (XXH3 checksum added automatically)
  tdtpcli --export orders --compress --output orders.tdtp.xml

  # Export archive-quality: kanzi max + large packet for broker (checksum automatic)
  tdtpcli --export kadrovye_prikazy --compress --compress-algo kanzi --compress-level 7 --packet-size 8 --output archive.tdtp.xml

  # Export with read-only fields (timestamp, computed, identity)
  tdtpcli --export employees --readonly-fields --output employees.tdtp.xml

  # Export with v1.4 integrity hashes (xxh3_128 on Schema + Data + Packet)
  tdtpcli --export users --compress --integrity --output users_v14.tdtp.xml

  # Export v1.4 + register fingerprint in xzMercury
  tdtpcli --export orders --compress --integrity \
    --mercury-url http://mercury:3000 --mercury-caller svc-exporter --output orders_v14.tdtp.xml

  # Export compact v1.3.1 (fixed fields written once per group)
  tdtpcli --export dept_emp --compact --fixed-fields dept_id,dept_name --output dept_emp.tdtp.xml

  # Auto-detect fixed fields by _prefix convention
  tdtpcli --export staffing_report --compact --fixed-fields _ --output staffing.tdtp.xml

  # Convert existing file to compact format
  tdtpcli --to-compact large.tdtp.xml --output large_compact.tdtp.xml

  # NOTE: export uses parallel packet processing — packet ORDER in multi-part
  #       files is not guaranteed. Use --test to verify integrity before import.

  # Convert TDTP to CSV (stdout)
  tdtpcli --to-csv users.tdtp.xml

  # Convert to CSV with output file
  tdtpcli --to-csv users.tdtp.xml --output users.csv

  # Convert with semicolon separator (Excel European locale)
  tdtpcli --to-csv orders.tdtp.xml --delimiter ';' --output orders_eu.csv

  # Convert with tab separator (TSV)
  tdtpcli --to-csv data.tdtp.xml -d '\t' --output data.tsv

  # Convert with UTF-8 BOM (required for Excel to open without import wizard)
  tdtpcli --to-csv users.tdtp.xml --bom --output users_excel.csv

  # Convert Cyrillic data to Windows-1251 (legacy tools)
  tdtpcli --to-csv erp.tdtp.xml --cp 1251 --output erp_win.csv

  # Convert compressed TDTP to CSV (auto-decompresses)
  tdtpcli --to-csv users_zstd3.tdtp.xml --output users.csv

  # Convert v1.4 integrity packet to CSV (runs security pre-flight first)
  tdtpcli --to-csv users_v14.tdtp.xml --output users.csv
  tdtpcli --to-csv users_v14.tdtp.xml --mercury-url http://mercury:3000 --output users.csv

  # Convert with column projection (select only needed fields)
  tdtpcli --to-csv orders.tdtp.xml --fields id,status,total --output orders_slim.csv

  # Convert with TDTQL filters
  tdtpcli --to-csv users.tdtp.xml --where 'balance > 1000' --order-by 'balance DESC' --output top_users.csv

  # Combine filters + projection + encoding
  tdtpcli --to-csv users.tdtp.xml --where 'is_active = 1' --fields id,name,email \
    --delimiter ';' --bom --output active_users.csv

  # Preview TDTP file in browser (HTML viewer)
  tdtpcli --to-html customers.tdtp.xml --open

  # View specific row range (rows 100-500)
  tdtpcli --to-html customers.tdtp.xml --row 100-500 --open

  # View first 100 rows
  tdtpcli --to-html orders.tdtp.xml --limit 100 --open

  # View last 50 rows (tail mode)
  tdtpcli --to-html logs.tdtp.xml --limit -50 --open

  # Combine range and tail: last 20 rows from range 100-500
  tdtpcli --to-html data.tdtp.xml --row 100-500 --limit -20 --open

  # Export directly to Excel
  tdtpcli --export-xlsx orders --output orders.xlsx

  # Convert TDTP to XLSX
  tdtpcli --to-xlsx orders.tdtp.xml --output orders.xlsx --sheet Orders

  # Import XLSX to database
  tdtpcli --import-xlsx orders.xlsx --strategy replace

  # Check file BEFORE import: integrity + row count + checksum
  #   --test: decompresses in memory, verifies XXH3 checksum (automatic when --compress was used),
  #           counts rows vs header, checks all multi-part siblings present.
  #   WHAT YOU LEARN: 'is this file intact and complete?'
  tdtpcli --test users.tdtp.xml

  # Inspect file BEFORE import: schema, field types, row count (no DB needed)
  #   --inspect: reads XML metadata, prints YAML summary of schema + header.
  #   WHAT YOU LEARN: 'what is in this file — table name, field names, types?'
  tdtpcli --inspect users.tdtp.xml

  # Recommended pre-import workflow:
  tdtpcli --inspect delivery.tdtp.xml   # understand structure
  tdtpcli --test    delivery.tdtp.xml   # verify integrity
  tdtpcli --import  delivery.tdtp.xml   # load to DB

  # Import from TDTP file
  tdtpcli --import users.tdtp.xml --strategy replace

  # Import MS Access export with exotic field names (%, spaces, #, etc.)
  tdtpcli --import access_export.tdtp.xml --clear --strategy replace

  # Import Russian ERP export (Cyrillic field names) — transliterate + clean
  tdtpcli --import erp_orders.tdtp.xml --translit --clear --strategy replace

  # Import European data (diacritics only, no special symbols)
  tdtpcli --import eu_data.tdtp.xml --translit --strategy replace

  # Export to RabbitMQ
  tdtpcli --export-broker orders --config rabbitmq.yaml

  # Export to Kafka with kanzi compression (4× less traffic than uncompressed)
  tdtpcli --export-broker users --compress --compress-algo kanzi --compress-level 6 --config kafka.yaml

  # Import from RabbitMQ / Kafka to database
  tdtpcli --import-broker --config rabbitmq.yaml
  tdtpcli --import-broker --config kafka.yaml --strategy replace

  # Import from broker — save as TDTP files instead of importing to DB
  #   Multi-part: users_part_1_of_5.tdtp.xml, users_part_2_of_5.tdtp.xml, ...
  tdtpcli --import-broker --output users.tdtp.xml --config kafka.yaml

  # Import from broker — save raw bytes (no parse/decompress)
  #   Exactly what the broker sent; naming: users_part_1_of_5.bin
  tdtpcli --import-broker --raw --output users.bin --config kafka.yaml

  # Streaming consumer daemon — Kafka only
  #   Listens indefinitely; terminated by Ctrl+C / SIGTERM
  #   Recommended for stable channels (LAN, dedicated WAN, 99.99%+ uptime)
  #   NOT recommended for unreliable links — use --import-broker (batch) instead
  tdtpcli --listen --config kafka.yaml --strategy replace

  # Incremental sync
  tdtpcli --sync-incremental orders --tracking-field updated_at

  # Execute ETL pipeline
  tdtpcli --pipeline etl-config.yaml

  # Execute pipeline with variables (@name=value — parametric pipelines)
  tdtpcli --pipeline dept_staff.yaml @dept=97-256
  tdtpcli --pipeline report.yaml @dept=97-256 @date_from=2025-01-01 @date_to=2025-12-31
  #   SQL:       WHERE col = '@dept'  → WHERE col = '97-256'   (text, quotes escaped)
  #              WHERE year = @year   → WHERE year = 2025      (numeric, bare)
  #   YAML dest: out/dept_{{dept}}.tdtp.xml → out/dept_97-256.tdtp.xml

  # Pipeline embeds used variables into output packet (PipelineContext v1.4)
  #   --inspect shows:  pipeline: dept-sync v2.1
  #                     pipeline_vars:
  #                       dept: sales
  tdtpcli --pipeline dept_staff.yaml @dept=sales

  # Import with precondition check: verify packet came from the expected pipeline run
  tdtpcli --config cfg.yaml --import data.tdtp.xml --expect-var dept=sales
  tdtpcli --config cfg.yaml --import-broker --expect-var dept=sales --expect-var region=EMEA

  # Execute pipeline in unsafe mode (allows custom SQL)
  tdtpcli --pipeline etl-config.yaml --unsafe

  # ETL pipeline with field name sanitization (YAML snippet):
  #   sources:
  #     - name: legacy_erp
  #       type: tdtp
  #       dsn: erp_export.tdtp.xml
  #       sanitize:
  #         translit: true   # "Имя" → "Imia"
  #         clear: true      # "Total %" → "Total_pct"

  # Inspect TDTP file metadata (no config needed)
  tdtpcli --inspect orders.tdtp.xml

  # Inspect live DB table: native types, FK relationships, row count, sample record
  #   --inspect-table: connects to DB, queries system catalog, prints YAML.
  #   Bracket-quoted names (MSSQL/Access style) are supported.
  tdtpcli --inspect-table Employees --config pg.yaml
  tdtpcli --inspect-table '[ZTR$Employee]' --config mssql.yaml
  tdtpcli --inspect-table "[dbo].[Orders]" --config mssql.yaml

  # Import to different table name
  tdtpcli --import users.xml --table users_backup

  # Compare two TDTP files
  tdtpcli --diff users-old.xml users-new.xml

  # Compare with custom key fields
  tdtpcli --diff old.xml new.xml --key-fields user_id --ignore-fields updated_at

  # Merge multiple TDTP files (union)
  tdtpcli --merge file1.xml,file2.xml,file3.xml --output merged.xml

  # Merge with intersection strategy
  tdtpcli --merge f1.xml,f2.xml --output common.xml --merge-strategy intersection

  # Merge with conflict resolution
  tdtpcli --merge old.xml,new.xml --output result.xml --merge-strategy right --show-conflicts

  # Export with data masking
  tdtpcli --export customers --mask email,phone

WORKING WITH VIEWS:

  tdtpcli supports database views for export/import operations:

  ✅ Export: Works with all views
  ✅ List: Use --list-views to see all views with updatable status
  ⚠️  Import: Works only with updatable views (marked with U*)

  # List all database views with updatable status
  tdtpcli --list-views --config config.yaml

  # Export from view (specify view name explicitly)
  tdtpcli --export my_view --output view_data.xml

  # Export from view with TDQL filters (e.g., by date)
  tdtpcli --export orders_daily --where 'date eq "09.02.2026"' --output orders.xml

  # Legend for --list-views output:
    U* = Updatable view (can import)
    R* = Read-only view (export only)

  AUTOMATION WITH VIEWS:

  Views allow powerful SQL-based data preparation combined with automatic exports.
  Create views with JOINs, aggregations, and filters, then export programmatically:

  # Check if required views exist before running export pipeline
  if tdtpcli --list-views | grep -q "R\*orders_summary"; then
    echo "View found, exporting..."
    tdtpcli --export orders_summary --output daily_report.xlsx
  fi

  # Export all read-only views automatically
  tdtpcli --list-views | grep "R\*" | while read line; do
    view=$(echo $line | sed 's/.*R\*//' | awk '{print $1}')
    tdtpcli --export "$view" --output "reports/${view}.xlsx"
  done

  # Setup test views in your database
  ./scripts/setup-views.sh sqlite test_data.db
  ./scripts/setup-views.sh postgres localhost 5432 postgres mydb

CONFIGURATION:

  Configuration files use YAML format. Create a sample config with:
    tdtpcli --create-config-pg > config.yaml

  Config structure includes:
    - database: Connection settings
    - broker: Message broker settings (optional)
    - resilience: Circuit breaker and retry settings
    - audit: Audit logging settings
    - processors: Data masking, validation, normalization

FEATURES:

  ✅ Database Adapters: PostgreSQL, MS SQL, SQLite, MySQL, Access (.mdb/.accdb, Windows)
  ✅ Message Brokers:
       MSMQ     — Legacy     (Windows-only, batch mode)
       RabbitMQ — Stability  (reliable delivery, batch mode)
       Kafka    — Speed      (ordered partitions, batch + streaming; production-ready)
  ✅ Broker Export: parallel compress+serialize, single-roundtrip SendBatch
  ✅ Broker Import: parallel decompression, in-order processing
  ✅ Raw Broker Save: --raw flag captures queue messages verbatim (no parse/decompress)
  ✅ Streaming Consumer: --listen (Kafka only) — stable channels only
  ✅ File Operations: Diff & Merge, CSV/HTML/XLSX conversion, Compact format
  ✅ CSV Export: --to-csv with delimiter, code page (utf8/1251/866), BOM, column projection
  ✅ XLSX Converter: Database ↔ Excel bidirectional
  ✅ Compression: zstd (1-19) + kanzi (6-7, 30% denser) + XXH3 checksum automatic with --compress
  ✅ v1.4 Integrity: xxh3_128 Schema+Data+Packet hashes + optional xzMercury registration (--integrity)
  ✅ Circuit Breaker: Protection from cascading failures
  ✅ Audit Logger: GDPR/HIPAA compliance
  ✅ Retry Mechanism: Exponential backoff with jitter
  ✅ Incremental Sync: Checkpoint-based synchronization
  ✅ Data Processors: Masking, validation, normalization
  ✅ TDTQL: SQL-like query language for filtering
